# MLB 預測勝率提升研究報告

本報告旨在研究並設計一套能有效提高 MLB 投注預測勝率的方法。報告將涵蓋數據源盤點、潛在新特徵研究、數據取得可行性分析，並最終提出一套完整的設計方案。

## 1. 現有數據源盤點

本節將分析現有的三個主要數據來源：MLB Stats API、`pybaseball` 套件，以及台灣運彩。

### 1.1 MLB Stats API

MLB Stats API 是美國職棒大聯盟官方提供的數據接口。它提供了非常全面和即時的比賽數據。

- **可取得數據**:
  - **比賽數據**: 即時比分、賽程、比賽狀態、場地資訊、天氣等。
  - **球員數據**: 個人基本資料、生涯統計數據（打擊、投球、守備）、單場表現。
  - **球隊數據**: 隊伍名單、排名、攻守統計數據。
  - **Play-by-Play 數據**: 比賽中每一次投球、每一次打擊的詳細記錄，包含球速、球種、進壘位置等。

- **優點**:
  - **權威性**: 官方數據，最準確、最即時。
  - **全面性**: 涵蓋數據種類非常廣泛，從高層次的賽程到最細節的 Play-by-Play 都有。

- **缺點**:
  - **文檔不全**: 官方沒有提供非常正式和完整的開發者文檔，許多 API 端點需要社群挖掘和維護。
  - **使用複雜**: 直接使用 API 需要處理原始的 JSON 格式數據，結構較為複雜。

### 1.2 pybaseball

`pybaseball` 是一個開源的 Python 套件，它封裝了多個公開的棒球數據源，極大地簡化了數據獲取的難度。

- **整合的數據源**:
  - **MLB Stats API**: 獲取官方的即時和歷史比賽數據。
  - **FanGraphs**: 提供大量的進階數據（Advanced Stats），如 wOBA, FIP, WAR 等。這是預測模型非常重要的數據來源。
  - **Baseball-Reference**: 提供豐富的歷史數據和球員資料。
  - **Statcast**: 提供最詳細的投打對決數據，包括擊球初速 (Exit Velocity)、擊球角度 (Launch Angle)、投手轉速 (Spin Rate) 等。這些是建立高精度模型的關鍵。

- **可取得數據**:
  - **Statcast 數據**: 可輕易獲取指定時間範圍內所有比賽的每一顆球的詳細物理數據。
  - **球員進階數據**: 可直接抓取 FanGraphs 上的球員賽季進階統計數據。
  - **賽程與賠率**: 可以獲取歷史賽程及部分盤口數據。
  - **預期統計數據 (Expected Stats)**: 如 xwOBA (預期加權上壘率)。

- **優點**:
  - **簡單易用**: 將複雜的數據抓取流程封裝成簡單的 Python 函數調用。
  - **數據整合**: 整合了多個最重要的數據源，無需自行分別處理。
  - **專為分析設計**: 提供的數據格式（通常是 Pandas DataFrame）非常適合直接用於數據分析和機器學習。

- **結論**: 在實作上，**強烈建議以 `pybaseball` 作為主要數據獲取工具**，它簡化了來自 MLB Stats API、FanGraphs 和 Statcast 的數據整合工作。


### 1.3 台灣運彩

台灣運彩是台灣合法的運動彩券發行機構，是獲取本地投注賠率數據的主要來源。

- **可取得數據**:
  - **即時盤口**: 提供未來賽事的讓分、不讓分（獨贏）、總分大小等盤口和對應的賠率。
  - **歷史賽果**: 網站上可以查詢已結束比賽的最終結果。
  - **歷史賠率**: **(需驗證)** 官方網站通常不直接提供歷史賠率的 API 或下載功能。獲取歷史賠率數據通常需要**自行編寫網頁爬蟲 (Web Scraper)** 定期抓取並存儲。

- **優點**:
  - **本地市場指標**: 賠率反映了本地市場的投注熱度和莊家對比賽結果的預期，本身就是一個極具價值的預測特徵（隱含機率）。
  - **目標明確**: 我們的目標是提高投注勝率，因此運彩的賠率是模型必須參考的基準線。

- **缺點**:
  - **無官方 API**: 數據獲取相對不便，需要額外開發和維護爬蟲程式。
  - **數據限制**: 僅提供盤口相關數據，沒有深入的比賽統計數據。

## 2. 潛在新特徵研究與可行性分析

本節將探討多個可能影響比賽結果的進階特徵，並分析其數據取得的可行性。

### 2.1 天氣數據

- **影響**: 溫度、濕度、風向、風速會顯著影響球的飛行距離（影響全壘打和長打），以及投手的控球能力。例如，在溫暖、乾燥、順風的環境下，更容易出現高分比賽。
- **數據取得**:
    - **可行性**: 高。
    - **方法**: 可透過免費或付費的第三方天氣 API 服務，如 `OpenWeatherMap`、`WeatherAPI` 等。只需獲取比賽場地的經緯度，即可查詢特定日期的歷史天氣數據。
    - **成本**: 大部分天氣 API 提供有限度的免費方案，對於每日 MLB 比賽查詢通常已足夠。

### 2.2 球場因素 (Ballpark Factors)

- **影響**: 每個棒球場的尺寸、牆高、界外區大小、海拔等都不同，導致其對得分、全壘打、安打等結果的影響也不同。例如，科羅拉多洛磯隊主場 Coors Field 因高海拔空氣稀薄，是著名的「打者天堂」。
- **數據取得**:
    - **可行性**: 非常高。
    - **方法**: 像 **ESPN**、**FanGraphs** 等體育數據網站每年都會計算並公佈各球場的球場因素。這是一個相對靜態的數據，每年更新一次即可。可透過 `pybaseball` 或直接從網站上抓取。
    - **成本**: 免費。

### 2.3 打者進階數據

- **影響**: 傳統的打擊率(AVG)、全壘打(HR)、打點(RBI)等數據無法完全反映打者的真實貢獻。進階數據能更準確地衡量打者的表現。
    - **wOBA (加權上壘率)**: 綜合評估打者的上壘貢獻，比傳統 OPS 更準確。
    - **xwOBA (預期加權上壘率)**: 根據擊球初速和角度計算，排除守備和運氣成分，更能反映打者真實的打擊能力。
    - **OPS+ (標準化攻擊指數)**: 將球員的 OPS 與聯盟平均水平、球場因素進行校正，便於跨時代、跨球場比較。
- **數據取得**:
    - **可行性**: 非常高。
    - **方法**: **`pybaseball` 套件可以直接從 FanGraphs 獲取這些數據**。
    - **成本**: 免費。

### 2.4 投手進階數據

- **影響**: 傳統的防禦率(ERA)會受到守備能力的影響。進階數據能更好地評估投手獨立的投球能力。
    - **FIP (費爾丁獨立投球)**: 只考慮投手能完全控制的結果（三振、四壞、觸身、全壘打），排除守備因素，比 ERA 更能預測投手未來表現。
    - **xFIP (預期費爾丁獨立投球)**: 在 FIP 基礎上，將全壘打率標準化為聯盟平均水平，進一步降低運氣成分。
    - **K/9 (每九局三振次數)**: 衡量投手的壓制力。
    - **BB/9 (每九局保送次數)**: 衡量投手的控球能力。
- **數據取得**:
    - **可行性**: 非常高。
    - **方法**: **`pybaseball` 套件可以直接從 FanGraphs 獲取這些數據**。
    - **成本**: 免費。

### 2.5 牛棚疲勞度

- **影響**: 牛棚（後援投手）的表現對比賽後半段的勝負至關重要。如果一個隊伍的牛棚在過去幾天頻繁出賽、投球數過多，他們的疲勞度會增加，表現下降的風險也會提高。
- **數據取得**:
    - **可行性**: 中等。這不是一個現成的數據點，**需要自行建模計算**。
    - **方法**:
        1. 使用 `pybaseball` 獲取過去 3-5 天的比賽 Play-by-Play 數據。
        2. 追蹤每位後援投手的出場日期和投球數。
        3. 建立一個疲勞指數模型，例如，綜合考慮近期的投球總數、是否連續出賽等因素，為每個隊伍的牛棚計算一個疲勞分數。
    - **成本**: 免費（但需要開發成本）。

### 2.6 主客場表現差異

- **影響**: 球員在主場和客場的表現通常存在差異（主場優勢）。這可能源於對場地的熟悉度、球迷支持、客場旅途勞累等。
- **數據取得**:
    - **可行性**: 非常高。
    - **方法**: 球員和球隊的主客場數據 (Home/Away Splits) 是標準統計數據。可透過 `pybaseball` 輕易獲取。
    - **成本**: 免費。

## 4. 設計方案

綜合以上研究，我們提出以下設計方案，旨在建立一個基於進階數據的、自動化的 MLB 比賽預測模型。

### 4.1 推薦的特徵工程方案

建議模型至少包含以下幾類特徵：

1.  **基礎特徵**:
    - **球隊近期表現**: 如近 10 場的勝率、平均得分、平均失分。
    - **主客場**: 標示比賽是在主場還是客場。

2.  **盤口特徵**:
    - **台灣運彩賠率**: 將不讓分賠率轉換為隱含機率 (Implied Probability)，作為市場預期的基準線。

3.  **先發投手特徵 (關鍵)**:
    - **進階數據**: 使用 `xFIP`, `K/9`, `BB/9` 作為衡量投手實力的核心指標，而非傳統的 `ERA`。
    - **對戰歷史 (H2H)**: 先發投手對戰對方球隊的歷史表現數據。
    - **近期狀況**: 投手最近 3-5 次先發的表現。

4.  **打者特徵**:
    - **團隊進階數據**: 使用球隊整體的 `wOBA`, `xwOBA`, `OPS+` 來評估打線的真實火力。
    - **左右打對戰優勢**: 根據對方先發投手的左右投，計算打線中左右打者的比例優勢。

5.  **衍生與環境特徵**:
    - **牛棚疲勞度**: 根據前述方法，建立一個量化指標。
    - **球場因素**: 將比賽場地的得分、全壘打等因素指數化後納入模型。
    - **天氣數據**: 將溫度、風速等天氣狀況作為特徵。

### 4.2 預期準確率提升

- **從基礎到進階**: 傳統的預測模型可能僅依賴勝率、防禦率等基礎數據。通過引入 `xFIP`, `xwOBA` 等進階數據，我們的模型能夠更好地排除運氣和守備的干擾，更接近球員和球隊的真實實力，**預期將顯著提升預測的穩定性和長期勝率**。
- **捕捉動態因素**: 納入牛棚疲勞度、天氣等動態變量，使模型能夠捕捉到更多影響單場比賽結果的細微因素，從而發現賠率未能完全反映的投注價值。
- **目標**: 長期目標不是追求 100% 的準確率，而是**找到機器模型預測機率與盤口隱含機率差異最大的「價值投注」機會**，從而實現長期穩定的正向回報。

### 4.3 實作優先順序

建議按照以下順序分階段進行開發與實作：

1.  **階段一：數據管道建立 (MVP)**
    - **目標**: 建立一個能自動獲取核心數據的管道。
    - **任務**:
        - 使用 `pybaseball` 建立每日自動腳本，獲取賽程、球隊統計、先發投手 FIP/xFIP、球隊 wOBA/xwOBA 數據。
        - 開發台灣運彩的網頁爬蟲，抓取每日比賽的賠率。
        - 將所有數據整合並儲存到結構化的資料庫或 CSV 檔案中。
    - **產出**: 一個可靠的、每日更新的數據集。

2.  **階段二：基礎模型搭建**
    - **目標**: 建立一個包含核心進階數據的預測模型。
    - **任務**:
        - 以先發投手的 xFIP 和球隊的 wOBA 為核心特徵，結合運彩賠率，建立一個基礎的分類模型（如邏輯回歸、XGBoost）。
        - 進行模型回測 (Backtesting)，評估其歷史預測表現。
    - **產出**: 第一個版本的預測模型及績效報告。

3.  **階段三：擴充特徵與模型優化**
    - **目標**: 引入更多環境和動態特徵，提升模型精度。
    - **任務**:
        - 整合天氣 API，加入天氣數據。
        - 開發牛棚疲勞度計算模型，並將其作為特徵。
        - 引入球場因素、主客場差異等數據。
        - 對模型進行調參和優化。
    - **產出**: 一個特徵更豐富、預測能力更強的優化模型。

4.  **階段四：自動化與監控**
    - **目標**: 實現整個流程的自動化運行和監控。
    - **任務**:
        - 將數據獲取、模型預測、結果輸出整個流程串聯成一個自動化任務。
        - 建立監控儀表板，追蹤模型的每日預測結果和長期投資回報率 (ROI)。
    - **產出**: 一個可投入日常使用的、自動化的 MLB 預測系統。
